# テスト計画書 — みせ勤

## 1. テスト戦略

- ユニットテスト: ビジネスロジック・ユーティリティ関数（Vitest）
- 統合テスト: API Routes・Server Actions（Vitest + テスト用DB）
- E2Eテスト: 主要ユーザーシナリオ（Playwright）

## 2. ユニットテスト対象

### business-day.ts
- 午前0時またぎのない勤務（通常勤務）の勤務日判定
- 深夜営業（23:00〜翌5:00）の勤務日判定
- 切替時刻ちょうど（境界値）
- タイムゾーンが異なる店舗の判定

### attendance.ts（勤務時間計算）
- 休憩なし勤務の実労働時間
- 複数休憩ありの実労働時間
- 日またぎ勤務の時間計算
- 退勤なし（勤務中）の部分計算

### time-clock.ts（状態遷移）
- 未出勤 → 出勤（正常）
- 勤務中 → 休憩開始（正常）
- 休憩中 → 休憩終了（正常）
- 勤務中 → 退勤（正常）
- 未出勤 → 休憩開始（エラー）
- 勤務中 → 出勤（エラー・二重出勤）
- 退勤済み → 出勤（エラー）
- 休憩中 → 退勤（確認必要）

### anomaly-detection.ts
- 退勤漏れ検出（12時間超）
- 休憩終了漏れ検出
- 逆転時刻（退勤 < 出勤）
- 複数店舗同時勤務
- 重複打刻（同一スタッフ・同一時刻）

### authorization.ts
- オーナーは全店舗アクセス可
- 全店舗管理者は全店舗アクセス可
- 特定店舗管理者は所属店舗のみアクセス可
- スタッフは自分のデータのみアクセス可
- 唯一のオーナーは権限降格不可

### wage-history.ts
- 適用期間内の時給取得
- 複数期間が重複しない
- 将来の期間は現在から取得しない

## 3. 統合テスト対象

### 認証
- 新規登録フロー
- メール認証フロー
- ログイン成功
- ログイン失敗（パスワード間違い）
- ログイン試行制限
- パスワードリセットフロー

### 組織・店舗
- 組織作成
- 店舗作成
- 店舗無効化（過去勤怠との関連保持確認）
- 打刻URL発行・再発行

### スタッフ管理
- スタッフ招待
- 店舗所属追加
- 時給設定
- 退職処理

### 打刻
- 出勤（正常）
- 二重出勤防止（冪等性）
- 別店舗同時勤務防止
- PIN連続失敗レート制限
- 休憩中退勤の自動休憩終了

### 勤怠・修正
- 管理者による勤怠修正（元データ保全確認）
- 修正申請作成
- 修正申請承認（勤怠への反映確認）
- 修正申請却下
- 修正申請後の監査ログ確認

### CSV
- 勤怠CSVエクスポート
- 監査ログCSVエクスポート
- 権限外データがエクスポートされないこと

### APIキー
- APIキー発行
- 発行後の読み取りAPIアクセス成功
- 無効化後のアクセス失敗
- スコープ外店舗のデータアクセス失敗

### 組織間分離
- 組織AのユーザーがBのデータにアクセスできないこと
- IDORテスト（他組織のIDを直接指定）

## 4. E2Eテストシナリオ（Playwright）

### シナリオ1: 初期設定
```
1. ユーザー登録
2. メール認証（モック）
3. オンボーディング: 組織名入力
4. オンボーディング: 店舗名・タイムゾーン入力
5. スタッフ追加
6. 店舗所属設定 + PIN設定
7. 打刻URLをコピー
8. 打刻URLにアクセスし店舗名が表示されることを確認
```

### シナリオ2: 通常勤務
```
1. 打刻URLにアクセス
2. スタッフ選択
3. PIN入力（正しいPIN）
4. 「出勤」ボタンをクリック
5. 完了画面の確認（スタッフ名・時刻）
6. 「休憩開始」をクリック
7. 「休憩終了」をクリック
8. 「退勤」をクリック
9. 管理画面にログインし、勤怠一覧で確認
```

### シナリオ3: 修正申請
```
1. スタッフとしてログイン
2. 勤怠一覧から対象の勤怠を選択
3. 修正申請フォームに入力（修正理由必須）
4. 申請完了確認
5. 管理者としてログイン
6. 修正申請一覧で申請を確認
7. 申請詳細で変更前後を比較確認
8. 承認ボタンをクリック
9. 承認完了確認
10. 勤怠詳細で修正が反映されていることを確認
11. 監査ログを確認
```

### シナリオ4: 権限確認
```
1. 店舗A限定管理者としてログイン
2. 店舗Aの勤怠一覧にアクセス → 表示される
3. 店舗Bの勤怠URLに直接アクセス → 403またはリダイレクト
4. 店舗Bのスタッフ詳細URLに直接アクセス → 403
```

### シナリオ5: 組織分離
```
1. 組織Aのオーナーとしてログイン
2. 組織Bの勤怠IDを直接URLに指定 → 404または403
3. 組織Bの店舗IDを直接URLに指定 → 404または403
```

## 5. テスト環境

- 単体・統合テスト用DB: PostgreSQLテストDB（別スキーマ）
- E2Eテスト: ローカルサーバー起動後にPlaywright実行
- CI: GitHub Actionsで自動実行（PR時）
